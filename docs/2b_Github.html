Github Introduction

Andreas Handel

Overview

This document gives a brief explanation of Github and how we will use it for this course.

Learning Objectives

  • Know what Git and Github are.
  • Know why one might want to use them.
  • Have created and set up a Github account.
  • Have joined the class organization.

General Git and Github Information

What are Git and Github

Git is what is called a version control system. The main idea is that as you (and your collaborators) work on a project, the software tracks and records any changes made by anyone. Technically Github is distinct from Git, it in some sense is the interface and Git the underlying engine (a bit like RStudio and R). Since we will only be using Git through Github, I tend to not distinguish between the two and in the following refer to all of it as just Github. Note that other interfaces to Git exist, e.g. Bitbucket, but Github is the most widely used one.

Why use Git/Github

You want to use Github to avoid this:

Github gives you a clean way to track your projects. It is also very well suited to collaborative work. Historically, version control was used for software development. However, it has become broader and is now used for many types of projects, including data science projects.

To learn a bit more about Git/Github and why you might want to use it, read this article by Jenny Bryan.

What to (not) use Git/Github for

Github is ideal if you have a project with a fair number of files, most of those files are text files (such as code, LaTeX, (R)markdown, etc.) and different people work on different parts of the project.

Github is less ideal if you have a lot of non-text files (e.g. Word or Powerpoint) and different team members might want to edit the same document at the same time. In that instance, a solution like Google Docs, Word+Dropbox, Word+Onedrive, etc. might be better.

Some notes

Github has a lot of special terminology that takes getting used to. They have a handy page with short descriptions of some of the important terms. Some of the terms you will encounter soon Repository (also often called repo), User, Organization, Branch, Fork, Push, Pull, Commit and Stage. For some reason, the last term is not explained in the list linked to above. It basically describes th step after you made changes and before you commit them, you need to stage them. It’s more or less an automatic (and in my opinion somewhat useless) extra step, but you need to know what it means.

Github is an alternative to systems such as Dropbox. You can work locally on different machines and everything is backed up in the cloud (on github.com). The (sometimes annoying) difference to Dropbox is that the syncing between your local computer and Github.com is not automatic. So don’t forget to pull before you start work on a repository and push once you are done!

Because Github works similar to Dropbox, I have heard/read people say that it is a bad idea to locally store your Github repositories in a folder that is synced by Dropbox. If you do, the Dropbox sync process and the Github sync process can conflict, leading to a mess. So for any github repository, store it locally in a folder that is not synced across your computers by another software (such as Dropbox, OneDrive, Box, etc.)
